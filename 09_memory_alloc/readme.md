# 1. Virtual Memory

Early computers had very limited and expensive RAM.
Even though CPUs could theoretically address more memory, the physical RAM installed was often much smaller.

Without virtual memory, programs directly accessed physical RAM. This caused several major problems:
*   Not Enough Memory
*   Memory Fragmentation
*   Sequrity and isolation

### 1.1 Not Enough Memory
*   A 32-bit CPU can address up to 2³² = 4 GB of memory.
*   older machines had far less physical RAM (e.g., 1–2 GB).
*   If a program accessed an address that didn’t physically exist, it would crash.
![](img/NotEnoughMemory.gif)

### 1.2 Memory Fragmentation
*  Consider a system with 4 GB of RAM:
    * Program A - 1 GB
    * Program B - 2 GB
    * Program C - 2 GB

1. Program A & Program B run :
2. Program A exits
3. Even though 2 GB is free in total, it is not contiguous, so a new program C requiring 2 GB cannot start.
![](img/MemoryFragmentation.gif)

### 1.3 Security and Isolation
*   If all programs share the same address space:
*   Two programs can accidentally write to the same memory address
*   This leads to data corruption, crashes, or security issues
![](img/SecurityandIsolation.gif)

# 2. The Key Idea of Virtual Memory
The core idea is simple:
**Each program gets its own private memory space**, called virtual memory.  
**Virtual memory is a layer of indirection**

* Programs work with virtual addresses
* RAM uses physical addresses
* A mapping exists between the two

![](img/maping.gif)

![](img/maping_1.gif)

![](img/maping.png)

This mapping is managed by the operating system and enforced by hardware.

## 3. Virtual Memory Terminology
| Term             | Meaning                                   |
| ---------------- | ----------------------------------------- |
| Virtual Memory   | Address space seen by a program           |
| Physical Memory  | Actual RAM hardware                       |
| Virtual Address  | Address generated by a program            |
| Physical Address | Address used to access RAM                |
| Page             | Fixed-size block of memory (usually 4 KB) |

## 4. Pages and Page Tables
Instead of mapping every byte individually:
* Memory is divided into pages (typically 4 KB)
* Each page is mapped using a page table

**Page Table**
* A page table maps:
    * Virtual Page Number (VPN) → Physical Page Number (PPN)
* Each program has its own page table

**Address Structure (4 KB pages)**
```
[ Virtual Page Number | Offset ]
```
* Offset = last 12 bits
* Offset stays the during translation

## 5. Solving the Original Problems
### 5.1 Not Enough Memory → Swap Space
* If RAM is full, pages can be stored on disk (swap)
* When accessed again:
    * A page fault occurs
    * OS loads the page back into RAM
This is slow, but better than crashing.

## 5.2 Memory Fragmentation → Non-Contiguous Mapping
* Virtual pages can map to any physical pages
* Physical memory does not need to be contiguous
* From the program’s perspective, memory is still continuous

## 5.3 Security → Isolation
* Each program has its own virtual address space
* Same virtual address in different programs maps to different physical memory
* Shared memory is still possible when explicitly configured (e.g., shared libraries)

## 6. Page Faults
A **page fault** occurs when:
* A program accesses a virtual page
* That page is not currently in RAM

**What Happens:**
1. CPU detects missing page
2. CPU raises a page fault exception
3. OS selects a page to evict (usually LRU)
4. Dirty pages are written back to disk
5. Requested page is loaded
6. Instruction resumes

Page faults are very slow, so minimizing them is critical for performance.

## 7. MMU – Memory Management Unit
The MMU is a hardware component that:
* Translates virtual → physical addresses
* Detects invalid accesses
* Generates page faults
The MMU is configured by the OS and operates transparently to programs.

**Solution: TLB**
* A small, fast cache inside the CPU
* Stores recent virtual → physical translations

**Key Points:**
* TLB hits are extremely fast (often < 1 cycle)
* Modern CPUs have:
    * Instruction TLB
    * Data TLB
* Entries are evicted using policies like LRU

## 9. Multi-Level Page Tables
Single-level page tables are too large.

![](img/Iz7Ti.png)

## Multi-Level Approach:
* Page table entries are split across multiple levels
* Only required portions are kept in RAM
* Unused parts can be swapped out

Example:
* 64-bit virtual address
* 8 KB pages
* VPN split into multiple indexes for different levels

Modern Linux systems use up to 5-level page tables on 64-bit systems.

10. Summary
* Each program runs in its own virtual memory space
* Memory is divided into pages
* Page tables map virtual pages to physical pages
* TLB speeds up address translation
* Page faults handle memory that is not in RAM
* MMU enforces memory protection and isolation
* Virtual memory enables:
    * Efficient memory usage
    * Process isolation
    * Security
    * System stability

# Linux Kernel Memory Allocation
## How This Fits with Virtual Memory
* **Kernel code also runs in a virtual address space**, just like user programs — but with special rules.

Important Differences (User vs Kernel)
| User Space                 | Kernel Space                      |
| -------------------------- | --------------------------------- |
| Per-process virtual memory | Global kernel virtual memory      |
| Can page out freely        | Most allocations must stay in RAM |
| Crashes affect one process | Bugs crash the entire system      |

**Even inside the kernel, we don’t touch RAM directly — we still work through virtual memory.**

## Kernel Memory Zones (Why GFP Flags Exist)[]
Physical memory is divided into zones.  
These exist because some hardware has limitations.

**Main Zones**
| Zone    | Purpose                         |
| ------- | ------------------------------- |
| DMA     | Old devices (24-bit addressing) |
| DMA32   | 32-bit DMA-capable devices      |
| NORMAL  | Most kernel allocations         |
| HIGHMEM | User memory (32-bit systems)    |

* When you allocate memory, you are asking the kernel to pick memory from the right zone.


## `kmalloc()` – Physically Contiguous Memory
```c
void *buf = kmalloc(1024, GFP_KERNEL);
```
* Physically contiguous 
* Fast 
* DMA Safe
* Limited size 
* Can fail under fragmentation

### Why Contiguous Matters
* Required for DMA
* Required for some hardware registers

![](img/kmaloc.png)

**kmalloc is like asking for one solid block of RAM — no breaks allowed.**

## kzalloc() – kmalloc + Zeroing
```c
struct my_dev *dev = kzalloc(sizeof(*dev), GFP_KERNEL);
```
### What it does
* Same as kmalloc
* Memory is zero-initialized

### Why drivers love it
* Avoids uninitialized memory bugs
* Safer for structures

*Use `kzalloc()` for structs, `kmalloc()` for raw buffers.*

<br>

## `vmalloc()` – Virtually Contiguous Memory

```c
void *buf = vmalloc(4 * 1024 * 1024);
```
* Large mamory allocation
* Virtually contiguous
* Physically fragmented
* Slower than `kmalloc`
* Not DMA-safe

![](img/vmalloc.png)

**vmalloc works because of paging. Virtual memory hides physical fragmentation.**

## kmalloc vs vmalloc (Driver Perspective)
| Feature             | kmalloc | vmalloc |
| ------------------- | ------- | ------- |
| Physical contiguity | Yes     | No      |
| Virtual contiguity  | Yes     | Yes     |
| DMA safe            | Yes     | No      |
| Performance         | Fast    | Slower  |
| Large buffers       | Limited | Good    |

**Driver rule**     
* If hardware touches it → kmalloc()
* If CPU-only buffer → vmalloc()

## GFP Flags – Allocation Context Matters
Common Flags
| Flag       | Meaning                  |
| ---------- | ------------------------ |
| GFP_KERNEL | Can sleep, most common   |
| GFP_ATOMIC | No sleeping (interrupts) |
| GFP_NOWAIT | Fail immediately         |
| GFP_DMA    | Allocate from DMA zone   |

Critical driver rule

* Never use GFP_KERNEL in interrupt context
* Use GFP_ATOMIC instead


## Slab Allocator – Fast Object Allocation
**Why Pages Alone Are Not Enough**  
The kernel already knows how to allocate pages (usually 4 KB).  
That works great for large memory needs, but kernel data structures are often tiny:
| Kernel Object  | Typical Size      |
| -------------- | ----------------- |
| Credentials    | ~16 bytes         |
| Mutex / Lock   | tens of bytes     |
| Thread / Task  | hundreds of bytes |
| Process struct | ~1 KB             |

🚫 Using a full 4 KB page for each small object would:
* Waste huge amounts of memory
* Be very slow
* Cause internal fragmentation

**So the kernel needs a memory allocator `below` page level.**
### The Core Idea of the Slab Allocator
    A page should only contain objects of the same type.

This single idea solves many problems at once.
```
Page 1 → only mutexes
Page 2 → only inodes
Page 3 → only task_struct
```
✅ Same-size objects
✅ No external fragmentation
✅ Cache-friendly
✅ Easy to manage


## SLUB Allocator Overview (Kernel Heap)
From this point onward, all memory-allocation discussions refer specifically to the **SLUB** implementation of the Linux **slab** allocator.

The kernel heap is fundamentally different from user-space heap implementations such as `glibc malloc`. Because the design goals and constraints are different, the **terminology is also different**.  
For example, it would be incorrect to talk about *chunks* when describing `slab` allocation.    

Instead, the **SLUB** allocator is built around the concepts of `caches, slabs, slots, and objects`.

![](img/SLUB.png)


### 1. Caches — The Top-Level Concept
A **cache** is the highest-level abstraction in the SLUB allocator.
* Each cache manages objects of a single, fixed size
* Examples:
    * 256-byte objects
    * 512-byte objects
    * A specific kernel structure type (e.g. task_struct)

If the kernel frequently allocates objects of a given size, it creates a dedicated cache for that size. This allows allocation behavior to be tuned precisely for the object being stored.

Internally, each cache is represented by a `kmem_cache` structure.


### 2. Slabs — Memory Backing a Cache
A **slab** is a contiguous region of memory that belongs to a cache.
* A slab consists of one or more physical pages
* The slab is divided into **slots**
* Each slot is exactly the size managed by the cache    

For example:
* A 256-byte cache → slabs contain many 256-byte slots
* A 512-byte cache → slabs contain many 512-byte slots

*The term `slab allocator` comes from this idea: `caches manage slabs, and slabs contain slots`.*

### 3. Slots and Objects
A **slot** is simply a fixed-size region of memory inside a slab.
* When a slot is free → it is available for allocation
* When a slot is allocated → it contains a **kernel object**

When `kmalloc()` returns a pointer, that pointer:
* Points into a slab
* Specifically refers to a slot
* That slot is now considered to hold an object

Once allocated, the memory in that slot is exclusively used for that object type.

### 4. Object Reuse and Initialization Benefits
One major advantage of **slab** allocation is **object reuse**.

In user-space:
* `malloc()` followed by `free()` gives no guarantees about future contents

In SLUB:
* A cache stores only one object type
* Objects can be pre-initialized
* When reused, much of the object’s state may already be valid

This significantly improves:
* Performance
* Cache locality
* Predictability

### 5. Internal SLUB Data Structures
At a technical level, a cache (`kmem_cache`) is divided into two major components:

**CPU-local data (`kmem_cache_cpu`):**
* Represents the working slab for a specific CPU
* Allows allocation without global locking
* Optimized for speed and low contention

**Node-level data (`kmem_cache_node`)**
* Tracks slabs not currently owned by a CPU
* Manages **partial slabs**
* Used when a CPU needs a new working slab


This design is similar in spirit to thread-local caches in user-space allocators, but instead of being per-thread, it is **per-CPU**.

### 6. Slab States
Each slab can be in one of three states:
| State   | Description                     |
| ------- | ------------------------------- |
| Full    | All slots allocated             |
| Partial | Some slots allocated, some free |
| Empty   | No slots allocated              |

Key behavior:
* Allocations prefer **partial slabs**
* This minimizes page usage
* Empty slabs may be kept or released depending on cache policy

### 7. Tracking and Performance Optimization
An important optimization in SLUB:
* **Fully allocated slabs are not tracked**
* The kernel does not need to know about them
* They reappear in metadata only when an object is freed

This reduces bookkeeping overhead and improves allocation speed.

Partial slabs, on the other hand, are tracked so they can be reused efficiently.

### 8. What to Focus On (For Driver Development)
For most kernel and driver work, the important mental model is simple:
* A **slab** is a contiguous memory region
* It contains fixed-size slots
* Each slot is either:
    * Free
    * Holding a kernel object

Understanding how objects are laid out inside slabs — and how slabs are reused — is far more important than memorizing internal data structures.

### 9. Relationship to Virtual Memory
The SLUB allocator **does not replace paging**.
Instead, it builds on top of it:
```
Virtual Memory Pages
        ↓
Page Allocator
        ↓
SLUB Allocator
        ↓
Kernel Objects
```
In Linux, almost all small kernel allocations ultimately come from SLUB-backed caches, including most `kmalloc()` calls.

**SLUB efficiently packs fixed-size kernel objects into pages, reduces fragmentation, and enables fast, scalable allocation on multicore systems.**
